# AgileClaw 🦾 — 프로젝트 개요 (Re-baseline)

## 한 줄 정의

> 목표(KPI)를 주면, 요청 목표별 Agile Team을 만들고 자동 루프를 돌리는 자율형 에이전트

## 리베이스 방향 (2026-03-02)

AgileClaw는 OpenClaw의 전체 플랫폼을 복제하지 않는다.  
대신 **NanoClaw의 철학(작은 코어 + 명확한 경계 + 코드 커스터마이징)** 을 채택한다.

핵심 선택:
- 코어는 작게 유지: 채널, 에이전트, 메모리, 스케줄러, 에자일 루프
- 확장은 코드로 분리: 스킬(`skills/*/SKILL.md`) + 도구 플러그인(`tools/*.py` 자동 로드)
- 목표 단위 운영: 요청 목표마다 Agile Team 차터(`memory/teams/team-*.md`) 생성/재사용
- 우선순위는 "기능 수"가 아니라 "목표 달성 루프 완결성"

## 왜 이 방향인가

OpenClaw는 매우 강력하지만 범위가 광범위하다.  
우리 목적은 모든 채널/플랫폼 지원이 아니라, Dave의 KPI 달성 자동화다.

따라서 AgileClaw는 아래만 코어로 유지한다.

1. Telegram (사람과의 보고/지시 채널)
2. Claude API (추론 엔진)
3. 도구 실행 계층 (shell/files/web/browser + 범용 KPI + 커스텀 도구)
4. 스케줄러 (자율 실행)
5. 에자일 루프 (계획→실행→측정→개선)

## 운영 모델

- 사용자가 목표를 요청하면 팀별 차터를 만들고 KPI 측정 방법부터 확정
- 크론 잡이 매일/매주 자동 실행
- 에이전트가 도구로 실제 수치를 측정
- 결과를 기반으로 다음 액션을 업데이트

## 확장 전략

코어에 기능을 계속 넣지 않는다.

- 새 실행 능력: `tools/*.py` 파일 추가
- 새 업무 프로토콜: `skills/<name>/SKILL.md` 추가
- 외부 연동(Threads/Reddit/Gmail 등): 선택형 도구로 추가 (코어는 플랫폼 비종속)

## 현재 단계

- Phase 0: 리베이스 (문서/구조/확장 포인트 확정)
- Phase 1: KPI 도구(Threads/Reddit) + 주간 자동 리포트
- Phase 2: 채널 확장/보고 자동화 고도화
